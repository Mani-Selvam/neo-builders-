# MASTER DEVELOPMENT PROMPT — MULTI-COMPANY CONSTRUCTION ERP ADMIN PANEL

Build a complete, production-ready, premium multi-company Construction ERP/Admin Panel named **NeoBuilder ERP**.

## 1. Recommended Technology Stack

Use the following stack:

* Frontend: React.js + Vite
* Backend: Node.js + Express.js
* Database: MongoDB + Mongoose
* Authentication: JWT Access Token + Refresh Token
* Password Security: bcrypt
* API Client: Axios
* Routing: React Router
* Icons: Lucide React or React Icons
* Notifications: SweetAlert2 or premium toast system
* File Upload: Multer or Cloudinary-compatible architecture
* Theme: CSS variables with complete Light Mode and Dark Mode
* Charts: Recharts
* State Management: Context API or Zustand
* Validation: React Hook Form + Zod/Yup or equivalent robust validation

The application must be fully responsive for:

* Desktop
* Laptop
* Tablet
* iPad
* Android mobile
* iPhone

Do not create a basic CRUD template. Build a polished SaaS-style ERP interface with premium UI/UX.

# 2. CORE MULTI-COMPANY ARCHITECTURE

This is a multi-tenant ERP system.

Each registered company is a completely isolated tenant.

The most important architecture rule is:

```text
Company A
 ├── Departments
 ├── Designations
 ├── Clients
 ├── Suppliers
 ├── Labourers
 ├── Employees
 ├── Expenses
 ├── Item Categories
 ├── Item UOMs
 ├── Items
 ├── Labour Types
 ├── Site Types
 ├── Sites
 ├── Vehicle Types
 ├── Trucks
 ├── Works
 └── Users

Company B
 ├── Departments
 ├── Designations
 ├── Clients
 ├── Suppliers
 ├── Labourers
 ├── Employees
 ├── Expenses
 ├── Item Categories
 ├── Item UOMs
 ├── Items
 ├── Labour Types
 ├── Site Types
 ├── Sites
 ├── Vehicle Types
 ├── Trucks
 ├── Works
 └── Users
```

Company A must never see, update, delete, search, export, print, count, or access Company B data.

Every tenant-owned database document must contain:

```js
companyId: {
  type: mongoose.Schema.Types.ObjectId,
  ref: "Company",
  required: true,
  index: true
}
```

Never trust `companyId` sent from the frontend.

Always derive `companyId` from the authenticated JWT/session:

```js
req.user.companyId
```

Every API query must be tenant scoped:

```js
Model.find({
  companyId: req.user.companyId
})
```

Every single-record fetch must be tenant scoped:

```js
Model.findOne({
  _id: req.params.id,
  companyId: req.user.companyId
})
```

Every update must be tenant scoped:

```js
Model.findOneAndUpdate(
  {
    _id: req.params.id,
    companyId: req.user.companyId
  },
  updateData,
  {
    new: true,
    runValidators: true
  }
)
```

Every delete must be tenant scoped:

```js
Model.findOneAndDelete({
  _id: req.params.id,
  companyId: req.user.companyId
})
```

This isolation rule must apply to:

* CRUD APIs
* Dashboard counts
* Search
* Filters
* Dropdowns
* Reports
* Exports
* Print views
* Analytics
* References
* Autocomplete
* Bulk operations
* File uploads
* Audit logs

Company signup must be simple and fast.

Do not ask for the full company profile during signup.

Show only:

* Company Name — required
* Contact Person — required
* Mobile No — required
* Email — required
* Password — required
* Confirm Password — required
* Accept Terms and Conditions — required

Optional:

* Country, only when needed for mobile number formatting

## On Successful Signup

Backend must automatically:

1. Generate Company ID
2. Generate unique Company Code
3. Create Company document
4. Create first Admin/Owner user
5. Assign `companyId` to the user
6. Assign role `Company Admin`
7. Set default subscription plan
8. Set subscription start date
9. Calculate expiry date
10. Set company status `Active`
11. Hash password
12. Create audit event
13. Return authentication tokens
14. Redirect to company dashboard

Example:

```text
Company Code: CMP-000001
```

Do not allow users to manually edit system-generated Company ID.

```js
{
  companyCode: {
    type: String,
    required: true,
    unique: true,
    index: true
  },

  companyName: {
    type: String,
    required: true,
    trim: true
  },

  gstNo: {
    type: String,
    trim: true
  },

  panNo: {
    type: String,
    trim: true
  },

  registrationNo: {
    type: String,
    trim: true
  },

  contactPerson: {
    type: String,
    required: true,
    trim: true
  },

  mobileNo: {
    type: String,
    required: true
  },

  email: {
    type: String,
    required: true,
    lowercase: true,
    trim: true
  },

  address: String,
  city: String,
  state: String,
  country: String,
  pincode: String,

  logo: {
    url: String,
    publicId: String
  },

  subscriptionPlanId: {
    type: mongoose.Schema.Types.ObjectId,
    ref: "SubscriptionPlan"
  },

  subscriptionStartDate: Date,
  subscriptionExpiryDate: Date,

  status: {
    type: String,
    enum: ["Active", "Inactive", "Suspended"],
    default: "Active"
  },

  profileCompleted: {
    type: Boolean,
    default: false
  },

  profileCompletionPercentage: {
    type: Number,
    default: 0
  },

  profilePromptDismissed: {
    type: Boolean,
    default: false
  },

  createdAt: Date,
  updatedAt: Date
}
```

Use Mongoose timestamps:

```js
{
  timestamps: true
}
```

Important architecture rule:

Do not store `subscriptionPlan` only as plain text if subscription management will grow.

Use:

```js
subscriptionPlanId
```

with reference to:

```text
SubscriptionPlan
```

This allows:

* Free plan
* Basic plan
* Pro plan
* Enterprise plan
* Feature restrictions
* User limits
* Site limits
* Storage limits
* Expiry management

# 17. SECURITY MIDDLEWARE

Recommended middleware order:

```js
router.get(
  "/",
  authenticate,
  checkCompanyStatus,
  checkSubscription,
  authorize("department", "view"),
  controller.getAll
)
```

Required middleware:

* authenticate
* tenantResolver
* checkCompanyStatus
* checkSubscription
* authorize
* validateRequest
* rateLimiter
* errorHandler

Never accept tenant ownership from request body.

Strip fields such as:

```text
companyId
createdBy
updatedBy
passwordHash
subscriptionPlanId
status
```

when the current role is not allowed to modify them.
